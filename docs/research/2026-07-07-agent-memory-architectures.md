# Memory architectures for personal AI assistants: state of the art, mid-2026

*Research report (2026-07-07) feeding MASTERPLAN.md Part II. Produced by a web-research agent. Scope: one user, years of data, fully local (SQLite + sqlite-vec + FTS5, Qwen3-30B-A3B interactive, big-model offline batch via Claude Code/Codex CLI). All claims cite sources; where a claim rests on synthesis rather than a fetched source, it says so. Note: one planned fetch (the full Hindsight paper HTML) was cut off by a session limit; its numbers come from search-result extracts and VentureBeat's coverage.*

## 1. Systems: mechanisms

### 1.1 Letta / MemGPT: self-editing context + sleep-time compute

MemGPT's core idea (2023) was OS-style memory hierarchy: the agent itself edits a small in-context "core memory" via tools, pages conversation history ("recall memory") and long-term storage ("archival memory") in and out like RAM vs. disk ([Letta agent-memory blog](https://www.letta.com/blog/agent-memory/), [legacy MemGPT docs](https://docs.letta.com/guides/legacy/memgpt-agents-legacy/)).

The significant recent evolution is **sleep-time compute** (2025): a *separate* background agent shares the primary agent's memory blocks and rewrites them asynchronously. The primary agent talks to the user but **cannot edit its own core memory**; the sleep-time agent holds the memory-editing tools and turns "raw context" into "learned context", clean, reorganized, deduplicated memory, during idle time. Frequency is configurable (higher frequency = more tokens = better-maintained memory), and Letta explicitly recommends a *stronger, slower model for the sleep-time agent* than for the interactive one ([Letta sleep-time compute blog](https://www.letta.com/blog/sleep-time-compute/), [sleep-time agents docs](https://docs.letta.com/guides/agents/architectures/sleeptime/)). The stated motivation: in original MemGPT, memory editing competed with conversation in one agent and "memories may become messy and disorganized"; async management improves both response latency and memory quality ([Letta blog](https://www.letta.com/blog/sleep-time-compute/)). Letta also rearchitected its whole agent loop in late 2025 ("Letta v1"), citing lessons from ReAct, MemGPT and Claude Code ([Letta v1 blog](https://www.letta.com/blog/letta-v1-agent)).

**This is the single most relevant architecture for Aitvaras**: "weak fast model interactive + strong slow model rewrites memory offline" is exactly the Qwen3-30B + nightly Claude Code split.

### 1.2 Mem0: extract-then-reconcile fact pipeline (+ optional graph)

Mem0 ([arXiv:2504.19413](https://arxiv.org/abs/2504.19413)) is a two-phase pipeline. **Extraction:** an LLM reads recent conversation turns (plus a rolling summary for context) and emits candidate discrete facts. **Update:** for each candidate, top-k similar existing memories are retrieved and an LLM "routing controller" classifies the operation as **ADD / UPDATE / DELETE / NOOP**: this is the reconciliation step that prevents duplicates and handles contradictions ([Mem0 paper](https://arxiv.org/html/2504.19413v1), [Dwarves breakdown](https://memo.d.foundation/breakdown/mem0)).

**Mem0ᵍ** adds a Neo4j-backed labeled graph (entity extraction, relation mapping, conflict invalidation). Reported results on LOCOMO: 26% relative improvement (LLM-as-judge) over the OpenAI memory baseline, **91% lower p95 latency and >90% token savings vs. full-context**: but the graph variant only beat base Mem0 by **~2% overall**, winning only on temporal and open-domain questions ([paper](https://arxiv.org/html/2504.19413v1), [Mem0 research page](https://mem0.ai/research)). That 2%-for-a-graph-database result is a key data point for §2.5.

### 1.3 Zep / Graphiti: bi-temporal knowledge graph with edge invalidation

Zep's Graphiti ([arXiv:2501.13956](https://arxiv.org/html/2501.13956v1), [GitHub](https://github.com/getzep/graphiti)) is the most complete treatment of *time* in agent memory:

- **Three subgraph tiers:** episodes (raw messages/JSON, kept non-lossy), semantic entities/edges (extracted facts), communities (label-propagation clusters with map-reduce summaries) ([paper](https://arxiv.org/html/2501.13956v1)).
- **Bi-temporal model:** every fact-edge carries `t_valid`/`t_invalid` (when true in the world) plus `t'_created`/`t'_expired` (when the system learned it/learned it was superseded) ([paper](https://arxiv.org/html/2501.13956v1), [Zep docs](https://help.getzep.com/graphiti/getting-started/overview)).
- **Edge invalidation:** on ingest, an LLM compares new edges against semantically related existing edges; a temporally overlapping contradiction sets the old edge's `t_invalid = new edge's t_valid`. Superseded facts are **invalidated, never deleted**, preserving history and enabling "as-of" queries ([paper](https://arxiv.org/html/2501.13956v1), [Neo4j blog](https://neo4j.com/blog/developer/graphiti-knowledge-graph-memory/)).
- **Entity resolution:** embedding + full-text search for duplicate candidates, then an LLM resolution prompt merges names/summaries ([paper](https://arxiv.org/html/2501.13956v1)).
- **Retrieval:** cosine + BM25 + breadth-first graph traversal, fused with RRF/MMR/cross-encoder rerankers ([paper](https://arxiv.org/html/2501.13956v1)).

Results: LongMemEval **71.2% vs 60.2% full-context (gpt-4o), with ~90% latency reduction (2.58s vs 28.9s) and 1.6k vs 115k context tokens**; biggest wins on temporal reasoning, multi-session synthesis, preference inference, but it *lost* accuracy on single-session questions vs. full context ([paper](https://arxiv.org/html/2501.13956v1)). The Zep authors themselves call the older DMR benchmark inadequate (60-message conversations, trivial fact lookups) ([paper](https://arxiv.org/html/2501.13956v1)).

### 1.4 LangMem: memory-type taxonomy + background manager

LangChain's LangMem SDK formalizes the now-standard **semantic (facts) / episodic (past interactions, often distilled into few-shot examples) / procedural (learned instructions, stored as prompt updates)** taxonomy, and splits memory formation into "hot path" (agent writes during chat) vs. "background" (a Memory Manager processes conversations afterwards, deciding store/update/delete and consolidating to avoid "memory hoarding") ([LangMem launch blog](https://www.langchain.com/blog/langmem-sdk-launch), [conceptual guide](https://langchain-ai.github.io/langmem/concepts/conceptual_guide/), [LangChain memory docs](https://docs.langchain.com/oss/python/concepts/memory)). Storage-agnostic (works over any store) ([DigitalOcean tutorial](https://www.digitalocean.com/community/tutorials/langmem-sdk-agent-long-term-memory)).

### 1.5 A-MEM: Zettelkasten-style atomic notes with memory evolution

A-MEM (NeurIPS 2025, [arXiv:2502.12110](https://arxiv.org/abs/2502.12110), [code](https://github.com/WujiangXu/A-mem)) stores each memory as an atomic note with LLM-generated contextual description, keywords, and tags; on insert it searches for related notes and creates links, and, its distinctive mechanism, **memory evolution**: adding a new note can trigger the LLM to *rewrite the contextual descriptions/tags of existing linked notes*, so old memories are reinterpreted in light of new ones ([paper](https://arxiv.org/html/2502.12110v1)). It beat SOTA baselines across six foundation models without a rigid graph schema, a middle path between flat lists and full KGs ([abstract](https://arxiv.org/abs/2502.12110)).

### 1.6 Generative Agents (Park et al. 2023): still the canonical loop

The Smallville paper ([arXiv:2304.03442](https://ar5iv.labs.arxiv.org/html/2304.03442)) remains the reference design everything above descends from:

- **Memory stream:** append-only natural-language observations (≈ Aitvaras's activity log).
- **Retrieval score = recency + importance + relevance** (normalized sum): recency = exponential decay (factor 0.995 per hour since last access), importance = LLM rating 1–10 at write time ("mundane → 1, extremely poignant → 10"), relevance = embedding cosine ([paper](https://ar5iv.labs.arxiv.org/html/2304.03442), [MemX glossary](https://memx.app/glossary/generative-agents/)).
- **Reflection:** triggered when the *sum of importance scores of recent events crosses a threshold* (~2–3×/day in practice); the agent asks the LLM "what are the most salient high-level questions about recent memories?", retrieves evidence per question, and writes higher-level insight memories *with pointers to the evidence*, building a reflection tree ([paper](https://ar5iv.labs.arxiv.org/html/2304.03442)).

### 1.7 Claude memory (Anthropic): summary document, project-scoped, user-editable

Timeline: search-triggered memory early 2025 → automatic memory for Team/Enterprise **Sept 12, 2025** (project-scoped memory, incognito mode) → all paid users **Oct 23, 2025** ([Anthropic announcement](https://www.anthropic.com/news/memory), [VentureBeat](https://venturebeat.com/ai/anthropic-adds-memory-to-claude-team-and-enterprise-incognito-for-all), [MacRumors](https://www.macrumors.com/2025/10/23/anthropic-automatic-memory-claude/)). Design signature: memory is a **generated, user-visible, directly editable memory summary**, scoped **per project** to prevent context bleed, plus incognito chats that never write memory ([VentureBeat](https://venturebeat.com/ai/anthropic-adds-memory-to-claude-team-and-enterprise-incognito-for-all), [MacRumors](https://www.macrumors.com/2025/10/23/anthropic-automatic-memory-claude/)).

Two 2025–26 developments matter here: (a) the **memory tool** on the API, a client-side, *file-based* memory (agent reads/writes memory files under a directory you host; pairs with automatic context editing), i.e. Anthropic's official pattern is "memory = files the agent edits" ([platform docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool), [Monigatti walkthrough](https://www.leoniemonigatti.com/blog/claude-memory-tool.html)); (b) as of May 2026 Anthropic is testing **"Memory Files"** in the consumer product, replacing the single rolling summary with **multiple topic-organized memory documents** the user can browse and edit, loading only relevant files per conversation to escape the single-summary size cap ([TestingCatalog](https://www.testingcatalog.com/anthropic-plans-claude-memory-update-with-new-memory-files/), [ClaudeAINews](https://www.claudeainews.com/news/anthropic-claude-memory-files-update)). Claude Code's CLAUDE.md/auto-memory is the same philosophy: plain-text, user-ownable memory files.

### 1.8 ChatGPT memory: two layers: editable facts + opaque profile

Since April 10, 2025, ChatGPT has two independent layers ([OpenAI Memory FAQ](https://help.openai.com/en/articles/8590148-memory-faq), [OpenAI announcement](https://openai.com/index/memory-and-new-controls-for-chatgpt/)):

1. **Saved memories**: discrete facts, auto-saved by a classifier or on explicit "remember this", injected into the system prompt ("Model Set Context") with timestamps; user can view/delete each one ([reverse-engineering by Embrace The Red](https://embracethered.com/blog/posts/2025/chatgpt-how-does-chat-history-memory-preferences-work/), [reference-saved-memories FAQ](https://help.openai.com/en/articles/11146739-how-does-reference-saved-memories-work)).
2. **Reference chat history**: implicit recall over past conversations. Observed behavior suggests it is *not* live search but a continuously maintained **user profile + recent-conversation summaries** carried in context ([Embrace The Red](https://embracethered.com/blog/posts/2025/chatgpt-how-does-chat-history-memory-preferences-work/)). Notably, this layer is *not itemwise-editable*: you can only toggle it or delete chats, the most-criticized part of the design and a deliberate contrast with Anthropic's editable summary.

### 1.9 Last-12-months additions (mid-2025 → mid-2026)

- **Hindsight** (Dec 2025, [arXiv:2512.12818](https://arxiv.org/abs/2512.12818)): current **LongMemEval SOTA at 91.4%**; open source with MCP server. Organizes memory into **separate typed networks** (separating evidence, synthesized summaries, and *evolving beliefs/opinions*) with **retain / recall / reflect** primitives producing a structured, queryable, *auditable* memory bank with stable belief updates. With an open 20B backbone it lifts LongMemEval accuracy 39.0%→83.6%; biggest gains exactly where personal assistants hurt: multi-session (21.1→79.7), temporal reasoning (31.6→79.7), knowledge updates (60.3→84.6) ([paper](https://arxiv.org/html/2512.12818v1), [VentureBeat](https://venturebeat.com/data/with-91-accuracy-open-source-hindsight-agentic-memory-provides-20-20-vision)). A 20B backbone reaching 83.6% is direct evidence a 30B-A3B can run a serious memory system.
- **MemOS** (May–Jul 2025, [arXiv:2505.22101](https://arxiv.org/pdf/2505.22101), [GitHub](https://github.com/MemTensor/MemOS)): "memory operating system": a **MemCube** abstraction unifying plaintext, activation (KV-cache), and parametric memory with metadata (source, version, lifecycle) for scheduling and governance.
- **MIRIX** (2025): multi-agent memory with **six memory types** (Core, Episodic, Semantic, Procedural, Resource, Knowledge Vault) and a coordinator routing updates/retrieval ([survey coverage](https://arxiv.org/html/2603.07670v1)).
- **Surveys/benchmarks wave:** "Memory for Autonomous LLM Agents: Mechanisms, Evaluation, and Emerging Frontiers" (Mar 2026, [arXiv:2603.07670](https://arxiv.org/html/2603.07670v1)); "Memory in the Age of AI Agents" survey + paper list ([GitHub](https://github.com/Shichun-Liu/Agent-Memory-Paper-List)); curated lists ([TeleAI Awesome-Agent-Memory](https://github.com/TeleAI-UAGI/Awesome-Agent-Memory)); memory-security survey coining "mnemonic sovereignty" (Apr 2026, [arXiv:2604.16548](https://arxiv.org/html/2604.16548v1)); an architectural study of forgetting across 13 memory configurations (Jun 2026, [arXiv:2606.15903](https://arxiv.org/pdf/2606.15903)); adaptive per-task memory structures (Feb 2026, [arXiv:2602.14038](https://arxiv.org/pdf/2602.14038)); SAGE "novelty gate" to stop redundant memory writes (2026, [arXiv:2605.30711](https://arxiv.org/pdf/2605.30711)).
- **Ecosystem:** Cognee (OSS "memory control plane", embeddings+graph, ~17.6k stars) and hosted players (Supermemory, Memori, EverMind) ([vectorize comparison](https://vectorize.io/articles/best-ai-agent-memory-systems), [Cognee blog](https://www.cognee.ai/blog/guides/best-open-source-ai-memory-tools-for-llm-agents-and-developers)).

### 1.10 Local-first personal tools

- **screenpipe** ([GitHub](https://github.com/screenpipe/screenpipe), [architecture docs](https://docs.screenpi.pe/architecture)): Rust; **event-driven** 24/7 capture (OS events trigger screenshot + accessibility-tree walk at the same timestamp, instead of dumb interval recording); OCR/STT; **everything lands in local SQLite**, searchable, with plugins feeding agents. The most relevant capture-layer design: append-only sensory log in SQLite, interpretation layered on top, structurally identical to Aitvaras's activity log.
- **Rewind → Limitless**: Rewind (2022): always-on screen capture + OCR + search; 2024 pivot to the Limitless pendant (wearable conversation capture, cloud); **Meta acquired Limitless in Dec 2025 and the Rewind Mac app was shut down** ([screenpipe's history writeup](https://screenpipe.com/blog/rewind-ai-alternative-2026)). Lesson frequently drawn: cloud-tied personal capture dies with the company; local-first survives.
- **khoj** ([GitHub](https://github.com/khoj-ai/khoj)): self-hostable "second brain": pgvector embeddings over your documents, classic RAG + agents/automations. Memory = document index, not extracted facts ([HoangYell explainer](https://hoangyell.com/khoj-explained/)).
- **Charlie Mnemonic** (GoodAI, [GitHub](https://github.com/GoodAI/charlie-mnemonic)): open-source assistant with explicit **LTM + STM + episodic** split; stores user messages, responses, and environment feedback into LTM and integrates them into future context ([GoodAI intro](https://www.goodai.com/introducing-charlie-mnemonic/)). Early (2024) but one of the few *assistant-shaped* (not framework-shaped) open memory implementations.

## 2. Design patterns: what works

### 2.1 Discrete-fact extraction vs. whole-conversation summarization

**Who does what:** Mem0, LangMem, ChatGPT saved-memories, Zep's semantic edges = discrete facts. MemoryBank, ChatGPT's chat-history layer, Claude's memory summary = summaries/profiles. Zep, Hindsight, Generative Agents = **both, layered** (raw episodes kept non-lossy; facts/reflections derived).

**Evidence:** LongMemEval ([arXiv:2410.10813](https://arxiv.org/abs/2410.10813)) tested this directly: **fact-augmented key expansion** (indexing sessions by extracted facts, not just raw text) improves retrieval precision and answer accuracy, especially for knowledge updates and cross-session tracking; **session decomposition** (indexing at round granularity rather than whole-session summaries) improves token efficiency and multi-session reasoning. Summaries lose the detail needed for precise recall; raw-only loses the abstraction needed for cross-session questions. Mem0's ablation-by-architecture makes the same point from the other side: discrete facts + reconciliation beat both full-context and summarization baselines at a fraction of the tokens ([Mem0 paper](https://arxiv.org/html/2504.19413v1)). **Consensus 2026: extract atomic facts for precision, keep raw episodes for grounding, and maintain summaries only as a navigation/profile layer, never as the sole store.** Zep explicitly keeps episodes as the "non-lossy foundation from which semantic information derives" ([Zep paper](https://arxiv.org/html/2501.13956v1)).

### 2.2 Consolidation / reflection passes

- **Trigger/cadence:** Generative Agents reflect when summed importance of recent events crosses a threshold (~2–3×/day) ([paper](https://ar5iv.labs.arxiv.org/html/2304.03442)); MemoryBank consolidates **daily** (conversation → daily event summary → rolling global user profile, a two-level hierarchy) ([arXiv:2305.10250](https://arxiv.org/abs/2305.10250)); Letta sleep-time agents run on a configurable per-N-messages/idle-time schedule with an explicit token-cost vs. memory-quality dial ([Letta docs](https://docs.letta.com/guides/agents/architectures/sleeptime/)); LangMem runs a background manager after conversations ([conceptual guide](https://langchain-ai.github.io/langmem/concepts/conceptual_guide/)).
- **What the prompts do:** generate salient questions about recent memories, retrieve evidence, write insights *citing the evidence* (Generative Agents); rewrite/reorganize memory blocks into "clean, concise, detailed" learned context (Letta); merge duplicates, resolve contradictions, prune (LangMem "memory hoarding" avoidance); re-describe old notes in light of new ones (A-MEM memory evolution); revise beliefs stably and auditably (Hindsight reflect).
- **Use a stronger model offline:** explicitly recommended by Letta (fast model interactive, larger model for sleep-time memory work) ([blog](https://www.letta.com/blog/sleep-time-compute/)): the strongest published validation of the nightly-Claude-Code-batch plan.

### 2.3 Temporal validity and contradiction handling

Zep/Graphiti is the reference: **bi-temporal timestamps on every fact** (`t_valid`/`t_invalid` for world-time, `t'_created`/`t'_expired` for system-time), LLM-checked contradiction on ingest, and **supersession by invalidation, not deletion** ([Zep paper](https://arxiv.org/html/2501.13956v1), [Graphiti docs](https://help.getzep.com/graphiti/getting-started/overview)). Mem0 handles the same problem more cheaply with its UPDATE/DELETE routing over top-k similar memories, but loses history ([Mem0 paper](https://arxiv.org/html/2504.19413v1)). The payoff is measurable: temporal reasoning and knowledge-update questions are where naive RAG collapses and structured/temporal systems win big, Zep +18.5 pts overall with temporal reasoning among the largest gains ([Zep paper](https://arxiv.org/html/2501.13956v1)); Hindsight knowledge-updates 60.3→84.6 and temporal 31.6→79.7 ([paper](https://arxiv.org/html/2512.12818v1)); LongMemEval's **time-aware query expansion** (extract time bounds from the query, restrict candidate recall to that window) yields +7–11% on temporal questions ([LongMemEval](https://arxiv.org/abs/2410.10813)). **The bi-temporal columns are portable: nothing about them requires a graph database.**

### 2.4 Importance scoring, decay, forgetting

- **Importance at write time:** LLM-rated 1–10 (Generative Agents). Cheap, works, but only used as *one additive term* in retrieval, never as a hard filter.
- **Decay:** exponential recency decay on last-*access* (not creation): Generative Agents 0.995/hour ([paper](https://ar5iv.labs.arxiv.org/html/2304.03442)); MemoryBank implements Ebbinghaus-style strength: each recall reinforces a memory's strength, unrecalled memories decay ([arXiv:2305.10250](https://arxiv.org/abs/2305.10250)). At a companion timescale MemoryBank reported this sufficed to keep the store coherent; follow-on work continues to build on the Ebbinghaus framing ([ACM ICICT 2026 paper](https://dl.acm.org/doi/full/10.1145/3803291.3803294)).
- **Forgetting = demotion, not deletion:** production systems converge on never hard-deleting automatically, Zep invalidates with full audit trail; ChatGPT/Claude only delete on user action ([OpenAI FAQ](https://help.openai.com/en/articles/8590148-memory-faq), [MacRumors](https://www.macrumors.com/2025/10/23/anthropic-automatic-memory-claude/)). Recent academic work warns that *where* forgetting logic sits changes behavior materially across architectures ([arXiv:2606.15903](https://arxiv.org/pdf/2606.15903)), and gating *writes* for novelty (don't store near-duplicates) is emerging as the cheaper lever than gating deletes ([SAGE, arXiv:2605.30711](https://arxiv.org/pdf/2605.30711)).

### 2.5 Flat list vs. entity-centric records vs. full knowledge graph: at personal scale

The honest reading of the evidence:

- **Full KG:** Mem0ᵍ bought ~2% overall over flat Mem0, only winning temporal/open-domain, at the cost of Neo4j plus much heavier extraction/update phases ([Mem0 paper](https://arxiv.org/html/2504.19413v1)). Zep's wins are real but come bundled with per-ingest LLM entity resolution, community detection with periodic full recomputes, and a graph DB, machinery motivated by *enterprise* multi-user scale ([Zep platform page](https://www.getzep.com/platform/graphiti/)).
- **Structured-but-not-graph is the current SOTA:** Hindsight's typed memory networks (facts vs. episodes vs. observations vs. beliefs) with good retrieval and reflection beats every graph system on LongMemEval ([arXiv:2512.12818](https://arxiv.org/abs/2512.12818), [VentureBeat](https://venturebeat.com/data/with-91-accuracy-open-source-hindsight-agentic-memory-provides-20-20-vision)). A-MEM similarly gets linking benefits with atomic notes + LLM-created links, no schema ([paper](https://arxiv.org/html/2502.12110v1)).
- **Flat lists cap out:** ChatGPT's flat saved-memories list plus Claude's single rolling summary are both being outgrown by their own vendors, Anthropic's Memory Files test explicitly exists because one summary document hits context-pressure limits ([TestingCatalog](https://www.testingcatalog.com/anthropic-plans-claude-memory-update-with-new-memory-files/)).

**At personal scale (thousands of facts, one user), the sweet spot is entity-tagged atomic facts with bi-temporal validity, a "graph-lite": facts reference entities, entities have profile records, but skip formal edges, community detection, and graph traversal.** This is synthesis of the above sources, and matches what the Mem0ᵍ delta implies: the graph's only unique wins (temporal, multi-hop) are recoverable with bi-temporal columns + entity tags + time-aware retrieval ([LongMemEval](https://arxiv.org/abs/2410.10813), [Zep paper](https://arxiv.org/html/2501.13956v1)).

### 2.6 Hybrid retrieval vs. always-in-context profile document

These are complements, not rivals, every major system ships **both**:

- **Retrieval mix:** Zep = cosine + BM25 + graph BFS, fused by RRF/MMR + rerankers; Generative Agents = relevance + recency + importance sum; LongMemEval adds time-window restriction from the query. FTS5 + sqlite-vec is literally the BM25+cosine pair; add recency/importance terms and time filters and the full mix is there.
- **Always-in-context profile:** Letta core memory blocks ([agent-memory blog](https://www.letta.com/blog/agent-memory/)), ChatGPT's Model Set Context + user profile, Claude's per-project memory summary, MemoryBank's global profile.
- **When each wins:** the profile wins for *stable, high-frequency, cross-cutting* facts (identity, preferences, current projects, things needed in most turns, where retrieval would have to fire every time anyway and might miss). Retrieval wins for the *long tail* (specific events, one-off details, temporal questions): Zep's numbers show retrieval-based memory beats even a 115k-token full context on multi-session/temporal questions while losing slightly on single-session detail, i.e. neither "everything in context" nor "retrieve everything" dominates.

### 2.7 Memory write policies + review UX

- **Agent-written during chat (hot path):** MemGPT/Letta memory tools, ChatGPT "remember this", Claude API memory tool ([platform docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool)). Pro: captures explicit "remember X" instantly. Con: competes with the conversation for model attention and degrades memory quality, the stated reason Letta moved editing to sleep-time agents.
- **Pipeline-extracted after (background):** Mem0, Zep ingestion, LangMem background manager, ChatGPT's implicit profile. Pro: better quality, dedup/reconciliation, no latency cost. Con: memory lags the conversation.
- **Converged best practice:** hot path only for explicit user requests + a tiny scratch block; everything else extracted/consolidated in background.
- **Review/edit UX:** the 2025–26 direction is decisively toward **user-visible, user-editable memory**: ChatGPT itemwise memory manager, Claude's editable summary + project scoping + incognito, Anthropic Memory Files as browsable documents. Security literature adds a governance reason: unreviewable memory is an injection/poisoning surface, "mnemonic sovereignty" ([arXiv:2604.16548](https://arxiv.org/html/2604.16548v1)).

## 3. Active learning from the user: prior art for a "curiosity queue"

Genuinely under-built territory, good news for Aitvaras as a differentiator, and there is now direct academic framing:

- **The "proactivity gap" is now a named, benchmarked problem.** "Ask Now, Use Later: Benchmarking the Proactivity Gap in Long-Lived LLM Agents" (May 2026, [arXiv:2605.28108](https://arxiv.org/html/2605.28108v1)) measures exactly this use case: agents that only ever do *passive* personalization "preserve only information the user has already revealed, leaving unspoken but future-relevant preferences outside the agent's usable user model", i.e., agents should **acquire latent user information before it is needed**. The literature anchor for a curiosity queue.
- **GATE, LM-generated elicitation beats self-description.** "Eliciting Human Preferences with Language Models" (Li, Tamkin et al., ICLR 2025, [arXiv:2310.11589](https://arxiv.org/abs/2310.11589)) showed in preregistered human experiments that having the LM **generate free-form, open-ended questions** about the user's preferences produces user models more informative than user-written prompts or labels, *and* users found answering questions less mentally demanding than writing self-descriptions. Directly validates short Q&A sessions over "tell me about yourself" onboarding forms.
- **Training/prompting for good question selection:** "Teaching Language Models To Gather Information Proactively" (Jul 2025, [arXiv:2507.21389](https://arxiv.org/html/2507.21389v1)) targets eliciting information that is "missing, unknown, or unspoken" beyond mere ambiguity resolution; "Asking Clarifying Questions for Preference Elicitation" (Oct 2025, [arXiv:2510.12015](https://arxiv.org/html/2510.12015v1)) and "Training Proactive and Personalized LLM Agents" (Nov 2025, [arXiv:2511.02208](https://arxiv.org/html/2511.02208)) both stress the failure mode to engineer against: **question fatigue, ask essential questions, suppress unnecessary ones**, and adapt to the user's tolerance. "Modeling Future Conversation Turns to Teach Models to Ask Clarifying Questions" (ICLR 2025, [proceedings](https://proceedings.iclr.cc/paper_files/paper/2025/file/97e2df4bb8b2f1913657344a693166a2-Paper-Conference.pdf)) selects questions by expected value for future turns; "Sensing What Surveys Miss" (Feb 2026, [arXiv:2602.00880](https://arxiv.org/html/2602.00880v2)) models the user to decide *when* proactive support is welcome.
- **Product prior art, Dot (New Computer):** the clearest shipped example of interview-style memory building. Onboarding was an explicit Q&A ("What do you do for work?", "How do you spend a typical Sunday?"), then **continuous follow-up questions** dug deeper on expressed interests to build long-term memory; TechCrunch's review headline, "Dot's AI really, really wants to get to know you", captures both the power and the fatigue risk ([TechCrunch](https://techcrunch.com/2024/06/21/dots-ai-really-really-wants-to-get-to-know-you/), [product page](https://new.computer/dot), [Freethink profile](https://www.freethink.com/robots-ai/ai-companion)). Reviews were strongly positive on the *feeling* of being understood, with the recurring caveat that questioning must be paced.
- **Internal question generation already exists in the canon:** Generative Agents' reflection step literally starts by asking "given recent memories, what are the most salient questions we can ask?", it answers them from memory ([paper](https://ar5iv.labs.arxiv.org/html/2304.03442)). A curiosity queue is the natural extension: **when reflection generates a question the memory store cannot answer, route it to the user instead of the archive.** No shipped system was found that closes this exact loop (reflection → unanswerable question → queued user Q&A): the pieces exist separately (Dot's interviewing, GATE's elicitation, sleep-time reflection), but the combination appears genuinely novel as a product mechanism.

**Design lessons from this literature:** (1) generate questions during offline reflection, not mid-conversation; (2) score them by expected utility (how many pending tasks/memories would this answer unblock, the "use later" framing); (3) deliver in short, user-initiated or clearly-bounded Q&A sessions (GATE's low-effort finding, Dot's pacing caveat); (4) cap frequency and let answers write straight into the fact store with provenance "user-answered question".

## 4. Recommendations for Aitvaras

Grounded in the evidence above; tailored to one user, years of data, SQLite + sqlite-vec + FTS5, 30B-A3B interactive, big-model nightly batch.

### 4.1 Schema shape: layered "graph-lite" in SQLite

Four layers, mirroring Zep's tiers and Hindsight's typed networks without a graph DB:

1. **Episodes (exists)**: the append-only activity log with provenance stays the non-lossy ground truth.
2. **Facts**: atomic, LLM-extracted statements: `fact(id, text, entity_ids[], kind: preference|biography|event|procedure|belief, importance 1–10, valid_from, valid_to, created_at, superseded_by, source_episode_ids[], source: user_stated|extracted|user_answered_question)`. Bi-temporal columns + supersession-by-invalidation are Zep's mechanism flattened into a table; typed kinds follow LangMem/MIRIX/Hindsight taxonomies. Index in FTS5 and sqlite-vec; per LongMemEval, embed **fact-augmented keys** (fact text + entity names), not raw log lines.
3. **Entities**: lightweight records (person/place/course/system) with a short LLM-maintained summary; facts tag entities. This buys the useful part of entity-centricity; skip edges/communities: Mem0ᵍ's ~2% says the marginal value doesn't cover the complexity at this scale.
4. **Profile documents**: a small set of topic-scoped markdown files (identity & preferences; current projects/courses; environment/homelab; communication style), *regenerated from the facts table*, not hand-maintained. Follows Letta memory blocks, Claude's editable summary, and Anthropic's Memory Files direction. Keep them plain files on disk: user-inspectable, git-diffable, editable: the emerging Anthropic pattern and the right answer to memory-poisoning governance.

### 4.2 Consolidation cadence: nightly big-model "sleep", light online writes

- **Hot path (30B, during chat):** only (a) explicit "merk dir das" → immediate fact write flagged `user_stated`, and (b) a session scratch block. No online reorganization, Letta's stated reason for moving memory editing out of the interactive agent.
- **Nightly batch (Claude Code subprocess): the sleep-time agent:** (1) extract facts from the day's episodes; (2) Mem0-style reconcile each against top-k similar existing facts → ADD/UPDATE/SUPERSEDE/NOOP, setting `valid_to` instead of deleting; (3) Generative-Agents-style reflection, pose salient questions, write insight-facts citing evidence; (4) regenerate profile documents; (5) emit curiosity-queue candidates; (6) a novelty gate to skip redundant writes ([SAGE](https://arxiv.org/pdf/2605.30711)). Daily cadence matches MemoryBank and Generative Agents (2–3×/day threshold ≈ nightly for a single user's volume). Weekly, run a deeper pass (entity-summary refresh, dedup sweep, decay update): the "periodic full refresh" Zep needs for its communities has the same shape.
- Using the *stronger* model offline is explicitly the published recommendation (Letta), and Hindsight's 83.6% with a 20B backbone shows the interactive 30B is sufficient for the retrieval/answering side.

### 4.3 Retrieval mix

Score = weighted **BM25 (FTS5) + cosine (sqlite-vec) + recency decay on last-access + importance**, RRF-fused, the Generative-Agents triple plus Zep's dual-search, all native to the existing stack. Two cheap additions with outsized measured payoff: **time-aware query expansion** (extract a time window from the query, filter on `valid_from/valid_to`, +7–11% on temporal questions, LongMemEval) and **as-of filtering** (only currently-valid facts by default; superseded facts retrievable on explicit "what did I use to…" queries). Skip cross-encoder reranking initially; at thousands of facts, candidate sets are small enough that the 30B can rerank in-prompt.

### 4.4 Profile-in-context vs. retrieved memories

Replace the flat ≤30-item injected list with: **always in context** = the 2–4 profile documents (bounded, regenerated nightly, covering stable high-frequency facts) + top facts by importance×recency; **retrieved per turn** = everything else. Evidence: vendors are abandoning both extremes, flat lists and single summaries hit ceilings, while retrieval-only systems lose on in-session detail and stable preferences (Zep's single-session regression); ChatGPT and Claude both converged on profile-plus-recall. Make memory writes visible in the activity log and the profile files user-editable (existing provenance rules extend naturally; editable-memory UX is now table stakes).

### 4.5 Curiosity queue

Build it, nothing shipped closes this loop yet (§3). Concretely: nightly reflection emits questions it cannot answer from memory; store as `question(id, text, motivation, expected_value, status)`; score by expected downstream utility; surface in short bounded sessions (GATE: low user effort, high information yield); hard-cap unprompted questions per day (fatigue findings; Dot's pacing lesson); answers become facts with `source=user_answered_question` and retire the question. Decay unanswered questions, a stale curiosity queue is another form of memory hoarding.

### 4.6 What to skip

- **Neo4j/full KG, community detection, graph traversal**: evidence says marginal at this scale (Mem0ᵍ +2%); revisit only if entity-tagged facts + temporal columns demonstrably fail multi-hop questions.
- **Automatic hard deletion**: invalidate/demote instead; deletion is user-initiated only (industry consensus).
- **Ebbinghaus-style strength machinery beyond a recency term**: MemoryBank's decay is elegant but its measured benefit over simple last-access decay is not established at personal scale; start with the Generative-Agents recency term and add strength counters only if retrieval noise appears.
